ReportsFW Architecture Document

This document describes the architecture of the ReportsFW financial stock-plan dashboard application, including runtime layers, data flow, persistence strategy, and deployment considerations.

1. System Overview

ReportsFW is a Next.js 16 application with React client components and Node.js API routes. It supports three data sources with automatic fallback: PostgreSQL (preferred) Oracle (secondary) In-memory demo state (resiliency fallback)

2. Logical Architecture

Browser (User) -> Next.js App Router Pages (Dashboard + CRUD screens) -> API Routes (/api/dashboard, /api/companies, /api/employees, /api/stock-plans) -> Data Access Layer (src/lib/oracle.ts) -> PostgreSQL via pg (if configured and reachable) -> Oracle via oracledb (if configured and reachable) -> Demo dataset in memory (always available fallback)

3. Layer Responsibilities

Layer Main Files Responsibility
Presentation src/app/*, src/components/* Render dashboard and management views, trigger API calls, display state and feedback.
API src/app/api/*/route.ts Provide GET/POST/PUT endpoints for dashboard and entity operations.
Data Access src/lib/oracle.ts Encapsulate schema management, seed, reads, writes, and fallback routing.
Domain Model src/lib/sample-data.ts Define Company/Employee/StockPlan/Report types and baseline demo records.
Database Schema db/schema.sql Reference DDL for Oracle and initial seed merge statements.

4. Data Model

  • companies: issuer-level metadata (name, ticker, sector, headquarters)
  • employees: people assigned to companies with role and salary
  • stock_plans: plan settings (type, annual contribution, vesting, status)
  • financial_reports: periodic portfolio/target/risk snapshots

5. Key Runtime Flows

Dashboard Read Flow

  1. Client calls GET /api/dashboard.
  2. API invokes getDashboardData().
  3. Data layer tries PostgreSQL, then Oracle, then demo fallback.
  4. Response returns normalized payload with source, summary, and entity arrays.

Create/Update Flow

  1. Form submits to entity route with POST (create) or PUT (update).
  2. Route delegates to saveCompany/saveEmployee/saveStockPlan.
  3. Data layer executes upsert transaction (PostgreSQL or Oracle).
  4. On DB failure, operation is applied to in-memory demo state to keep UX functional.

6. Non-functional Characteristics

  • Availability: built-in fallback ensures UI remains operational even when DB is unavailable.
  • Consistency: per-request write transactions in DB-backed mode with commit/rollback behavior.
  • Performance: lightweight payloads and server-side no-store API responses for near-real-time updates.
  • Scalability: PostgreSQL connection pooling via pg.Pool.

7. Deployment Inputs

  • Node.js runtime for Next.js application.
  • Optional PostgreSQL env vars: POSTGRES_HOST, POSTGRES_PORT, POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD
  • Optional Oracle env vars: ORACLE_USER, ORACLE_PASSWORD, ORACLE_CONNECTION_STRING
